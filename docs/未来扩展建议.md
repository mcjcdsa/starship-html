# 星舰 Starship 展示站 · 未来扩展建议

> 基于当前工程（`index.html` 四页 SPA、`app.js` 时序推演、`monitor-server.js` NXSF 监测、飞行器/封路数据层）整理。  
> 目标读者：产品规划、后续开发维护者。

---

| 模块 | 已实现 | 待加强 |
|------|--------|--------|
| 主页 | 简介、Flight 12 卡片、天气/测试/视频（NXSF）、倒计时状态 | 多任务切换、历史任务归档入口 |

## 二、扩展原则

1. **数据与展示分离**：继续以 `data/*.json` + `STARSHIP_CONFIG` 为单一事实来源，页面只负责渲染。
2. **渐进增强**：保留无后端可运行（内嵌 `*.js`）；有 `monitor-server` 时自动拉新。
3. **发射日优先**：Flight 12 窗口过后，通过配置切换「下一发」而非改代码。
4. **非官方声明**：所有扩展需在页脚/关于页强化「非 SpaceX / NXSF 官方」。

---

## 三、短期建议（1～2 周，投入小、收益高）

### 3.1 内容与配置

- **下一发任务切换**：在 `config.js` 增加 `activeFlight: 12`，主页/任务/时序表/发射场一键切换 Flight 13+。
- **时区与文案统一**：封路 Primary 窗口、倒计时、时序表「北京时间」统一标注 CST/CDT 与 UTC 对照行。
- **观看链接扩展**：`watchLinks` 增加 YouTube 直播、NSF Live、SpaceX X 等，按「未开始 / 直播中 / 结束」显示不同按钮状态。
- **数据更新脚本**：`package.json` 增加 `npm run build:data`，一键执行 `parse-vehicles-3.js` 并同步 `nxsf-vehicles.js`、`closures.js`。

### 3.2 体验修补

- **本地预览服务**：增加 `npm run dev`（如 `npx serve .` 或 `http-server`），避免 `file://` 下 `fetch` 失败；README 写明推荐用法。
- **封路/飞行器搜索**：列表顶栏增加按编号（Ship 39、Booster 19）与状态筛选。
- **时序表**：节点 hover 显示完整备注；支持 URL hash 深链到某一事件（`#missions&e=12`）。
- **无障碍**：地图与倒计时补充 `aria-live`；对比度复核 WCAG。

### 3.3 监测与数据

- **监测面板**：`monitor-server` 成功时，将 GO%、窗口、封路 Primary 写回 `config` 预览（仍须人工确认后发布静态数据）。
- **starbase 解析增强**：从 beach-road-access HTML 解析 BEACH/Road 段落，自动填充 `closures.json`（失败则回退当前静态文案）。

## 四、中期建议（1～3 个月）

### 4.1 信息架构

```
主页
├── 当前任务（Flight N）概览
├── 发射倒计时（全局组件）
└── 快速入口

任务
├── 任务列表（历次 Flight）
├── 任务详情 /flight/12
│   ├── 倒计时 + GO%
│   ├── 发射场地图
│   ├── 飞行时序（可嵌入本页）
│   ├── 发射流程条
│   └── 相关飞行器（Ship/Booster 链接）

飞行器
├── 列表（现有）
└── 详情 /vehicle/ship-39
    ├── 参数、状态时间线
    └── 参与任务列表

封路
├── 海滩通行（BEACH）
├── 道路（Road Updates）
└── 历史关闭记录（可选）
```

### 4.2 功能模块

| 功能 | 说明 | 技术要点 |
|------|------|----------|
| **任务历史库** | 每次试飞一页：结果、视频、关键时间、RUD/回收结论 | 扩展 `missions.json`，模板化详情页 |
| **时序回放** | 任务结束后用录制时间轴回放，支持暂停/拖动 | 与 `currentSeconds` 解耦，增加 `replayMode` |
| **实时同步增强** | WebSocket 推送监测变更，减少轮询 | `monitor-server` 增加 `/ws` |
| **通知订阅** | 浏览器 Notification：T-1h、封路 Primary 开始前 | Service Worker + 用户授权 |
| **多语言** | 中/英切换（时序事件名、UI 壳层） | `data/i18n/zh.json`、`en.json` |
| **PWA** | 离线可读封路/飞行器/上次时序快照 | `manifest.json` + cache 静态资源 |
| **发射场增强** | 真实发射方位、禁飞区示意、可选风玫瑰 | MapLibre 图层或静态 GeoJSON |
| **直播嵌入** | 官方/NSF iframe 或链接卡片，发射日置顶 | 注意 CSP 与 embed 可用性 |

### 4.3 前端工程化

- 引入轻量构建（可选）：Vite 打包，保留原生 JS 模块 `import`，CSS 按页拆分。
- ESLint + Prettier；对 `parse-vehicles-3.js`、`getEventStatus` 等核心逻辑加单元测试。
- 组件化：倒计时、时序轴、封路卡片拆为独立模块，便于 Flight 13 复用。

## 五、长期建议（3 个月+）

### 5.1 数据与生态

- **统一数据层**：抽象 `DataService`（静态 / monitor API / 未来自建 API），页面不直接 `fetch` 路径字符串。
- **版本化发布**：每次发射前打 tag `flight-12-data`，站点展示「数据版本号」与更新时间。
- **社区贡献**：GitHub 议题模板「更正事件时间 / 新增飞行器」，PR 只改 JSON，CI 校验 schema。
- **可选后端**：若流量增大，用 Node/Serverless 定时抓 NXSF + starbase，前端只请求自己的 CDN JSON（降低对用户本机 `monitor` 依赖）。

### 5.2 进阶可视化

- **3D 轨迹示意**（非真实弹道）：起飞→分离→助推返回→星舰再入，简模动画，与 `currentSeconds` 联动。
- **对比视图**：两次 Flight 时序差异（节点时间偏移、取消步骤高亮）。
- **统计面板**：助推器捕获成功率、舰体溅落/再入统计（来自 `nxsf-vehicles` 聚合）。

### 5.3 运营与合规

- **关于 / 免责声明页**：数据来源、时区、误差说明、联系方式。
- **分析（可选）**：隐私友好的访问统计（Plausible / 自托管 Umami）。
- **SEO**：每任务详情独立 `title`/`description`；Open Graph 图（发射日自动生成分享图可选）。

## 六、技术债务优先处理

按建议修复顺序：

1. `monitor-server` 与静态 `closures.json` 结构已统一；补全 **starbase HTML 解析** 与错误降级 UI。  
2. **NXSF 监测失败**时，任务页显示明确「离线模式」横幅，避免静默使用旧数据。  
3. 时序 **滚动动画** 与 **任务进度条** 语义分离（用户已反馈过「全蓝」误解），文档化交互说明。  
4. 大型 JSON（63 飞行器）考虑 **分页或虚拟列表**，改善低端机滚动性能。  
5. MapLibre / 外链 CDN：提供 **国内可访问的瓦片镜像** 或静态截图回退。

## 九、成功指标（可量化）

- 发射日当天：首屏倒计时与时序 **&lt; 2s** 可交互（本地 serve + 内嵌数据）。
- 封路 Primary 窗口与 starbase 官方 **时间一致率**（人工抽检 1 次/发）。
- 移动端 **飞行器列表** 滚动 60fps（或虚拟列表后无明显卡顿）。
- 监测服务可用时，GO% / 窗口变更 **&lt; 5 分钟** 反映到页面（轮询或 WS）。

## 十、总结

当前站点已完成「**单次试飞的一站式信息面板**」雏形：时序推演、硬件名录、封路海滩、发射场地图与 NXSF 监测。下一阶段应聚焦 **配置化切换下一发、数据流水线、任务/飞行器详情页** 三块，再考虑回放、推送与工程化；长期可向「**星舰试飞档案库 + 发射日实时面板**」演进，并始终保持静态可部署、官方数据源可追溯。

如需落地某一条 P0/P1 项，建议从 **`activeFlight` 配置 + `npm run dev` + 任务详情页模板** 开始拆 issue 逐步实现。
